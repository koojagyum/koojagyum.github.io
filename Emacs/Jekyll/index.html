Github의 Jekyll 페이지 번역.

Github Page는 일반적인 HTML 컨텐츠 뿐만 아니라 Jekyll 이라는 간단한 블로그와 비슷한 정적 사이트 생성기를 지원한다. Jekyll을 사용하면 내용 복사 없이 사이트내 모든 페이지에 대해 적용되는 헤더와 풋터 영역을 쉽게 만들 수 있다. 그리고 몇 가지 개선된 템플릿 도 사용할 수 있다.

1 Using Jekyll

Github Page에서 저장소 내의 특정 이름의 브랜치에 컨텐츠를 푸시하게 되면 Jekyll가 동작한다. 예를 들어 username.github.io 저장소에서 유저 페이지는 master 브랜치를 사용하고, 프로젝트 페이지는 gh-pages 라는 브랜치를 사용할 수 있다. 일반적인 HTML 사이트는 모두 유효한 Jekyll 사이트이기 때문에 Jekyll을 사용하면서 HTML 파일이 변형되는 것에 대해 염려할 필요는 없다. Jekyll의 특징과 사용 방법에 대한 문서 페이지 가 있으니 참고해 보자. 당장 사용해 보기 위해 Jekyll 포맷의 파일을 커밋해 보는 것으로부터 시작한다.

2 Jekyll 설치하기

컨텐츠를 Github Page로 공개하기 전에 Jekyll을 컴퓨터에 설치하여 사이트가 어떻게 생겼는지 미리 살펴보고 문제가 없는지 검사해 보는 것이 좋다.

운좋게도 Jekyll을 컴퓨터에 설치하고 Github Page의 세팅을 조정하는 것은 간단하다. 이와 관련하여 Github Pages Gem 과 버전 의존성 페이지 에서 수고해 주셨다. Jekyll을 설치하기 위해 아래 몇 가지 사항들이 필요하다.

  1. Ruby - Jekyll은 Ruby 프로그래밍 언어를 필요로 한다. Mac 사용자라면 대부분 Ruby에 대한 지원이 될 것이다. 터미널 어플리게이션을 열어 ruby --version 을 실행시켜서 확인해 보자. Ruby 버전이 1.9.3 이나 2.0.0 으로 시작해야 한다. 그렇지 않으면 이 페이지 를 참고해서 Ruby를 설치한다.
  2. Bundler - Ruby 소프트웨어의 버전을 만드는 패키지 매니저이다. 만일 Github Page 사이트를 로컬에 생성한다면 버전을 붙이는 것이 좀 더 쉬워질 것이다. Bundler를 설치하려면 gem install bundler 커맨드를 실행한다.
  3. Jekyll - 주인공인 Jekyll 이다. Gemfile 이라는 파일을 하나 생성하여 gem 'github-pages'​ 라고 쓰고 저장소에 추가한다. 그리고 bundle install 명령을 실행하면 된다. 만일 #2 과정을 생략하고 넘어갔더라도 gem install github-pages 명령을 통해 Jekyll을 설치할 수 있다. 하지만 실행시 문제가 있을 것이다. 아래는 Gemfile 의 예제이다(저장소의 루트 디렉토리에 넣자). 
source 'http://rubygems.org'
gem 'github-pages'

3 Jekyll 돌려보기

Github Page 빌드 서버와 연동시키려면 Jekyll을 Bundler와 같이 실행시켜야 한다. http://localhost:4000 으로 접근 가능한 환경에서 저장소의 루트 디렉토리에서 bundle exec jekyll serve 명령을 실행시킨다(프로젝트 페이지를 사용하기 위해서는 gh-pages 브랜치로 스위칭되어 있어야 한다). 모든 Jekyll 명령어를 보려면 Jekyll 문서 페이지 를 참고한다.

4 업데이트하기

Jekyll 은 오픈 소스 프로젝트 이며 현재도 업데이트되고 있다. Github Page 서버가 업데이트되면 PC의 소프트웨어는 최신 버전이 아니게 된다. 따라서 로컬 사이트는 Github에 공개된 페이지와는 좀 다르게 보일 것이다. Jekyll을 항상 최신 버전으로 유지하려면 bundle update 명령을 실행시키면 된다(만일 위의 2번 단계를 건너뛰었다면 gem update github-pages 를 실행시킨다).

5 사용자 설정

_config.yml 파일을 생성하여 Jekyll 세팅 을 대부분 손볼 수 있다.

5.1 기본 설정

아래 기본 설정은 Github에서 만들어진 것이고 사용자가 _config.yml 파일을 새로 쓸 수도 있다. 

highlighter: pygments
github: [Repository metadata]

Repository metadata에 대해서는 Github Page의 remository metadata 항목 을 참조하기 바란다.

5.2 Configuration Overrides

아래와 같이 _conig.yml 파일을 덮어써보자. 

safe: true
lsi: false
source: your top-level directory

source 세팅을 수정하면 페이지 빌드가 정상적으로 안 될 수도 있다. Github Page는 저장소 루트 레벨의 소스 파일만 다루기 때문이다.

6 머릿글을 작성해 주세요

Jekyll은 모든 파일의 상단에 표시되는 머릿글이 정의된 파일이 따로 필요하다. 머릿글은 메타데이터들만 들어갈 수 있고 대시 세 개(—)의 쌍으로 사용한다. 

---
title: This is my title
layout: post
---

Here is my page.

머릿글을 없앨 수도 있지만 대시 세 개는 들어가 있어야 한다. 

---
---

Here is my page.

이 파일이 _posts 디렉토리 안에 들어있다면 대시 세 개는 생략할 수 있다.

Jekyll 문서 페이지 에서 더 많은 정보를 확인할 수 있다.

6.1 Troubleshooting

Github에 컨텐츠를 푸시했는데 Jekyll 사이트가 제데로 렌더링을 하지 못하면 로컬로 Jekyll을 돌려서 어떠한 파싱 에러가 나는지 분석해볼 수 있다. 분석을 위해서는 Github에서 사용하는 것과 동일한 버전의 Jekyll과 다른 플러그인들 이 필요하다.

로컬에서 Github Page에서와 같은 버전을 사용하는지 확인하려면 Jekyll이 설치된 이후부터 주기적으로 gem update github-pages 를 실행시켜서 확인해야 한다. 이와 관련하여 Github Page의 Gem 저장소 에서 추가적인 정보를 확인할 수 있다.

Github에 컨텐츠를 푸시하고 난 다음에 페이지가 빌드되지 않는다면 Troubleshooting GitHub Pages build failures 문서를 확인해 보자.

Jekyll 페이지에 문제가 있다면 다른 프로젝트와 동일한 이름을 사용하고 있는 카테고리가 있는지 확인해 보길 바란다. 경로가 겹치는 것이 문제가 될 수 있기 때문이다. 예를 들어 'resume' 이라는 블로그 포스트를 유저 페이지 저장소에 만들고 프로젝트 페이지(gh-pages 브랜치)에도 똑같이 'resume' 페이지를 만든다면 서로 충돌이 생길 것이다.

7 Jekyll 끄기

저장소의 루트 디렉토리에 .nojekyll 파일을 만들고 Github에 푸시하면 Jekyll 과 관련된 동작을 모두 멈추게 할 수 있다. 사이트에 밑줄 문자(_)로 시작하는 디렉토리가 있다면 Jekyll은 이런 디렉토리를 마지막 출력지로 복사하지 않는데 이런 경우에만 예외 처리로 사용할 수도 있다.

8 컨트리뷰트하기

Jekyll이 원하는 데로 동작하지 않는다면 언제든지 fork 해서 pull 요청해 주기를 바란다. 우리는 사용자들의 컨트리뷰션을 환영한다.
