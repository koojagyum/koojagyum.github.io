OpenGL wiki의 OpenGL Object 문서 번역.

OpenGL Object 는 몇 가지 상태값을 갖는 OpenGL의 구조체라고 보면 될 것 같다. 이것들이 Context에 바인딩되면 이들이 가진 상태값이 Context의 상태값으로 매핑된다. 그러므로 현재 Context의 상태값이 변화하면 이 객체에 반영되며 Context 상태에 영향을 받는 함수들도 이 객체의 상태값을 사용한다.

OpenGL은 "상태 머신" 이다. API 호출은 OpenGL의 상태를 변화시키고 어떤 상태값을 쿼리하여 가져오기도 하며, OpenGL이 무엇인가 렌더링할 때에는 현재 상태를 참고하기도 한다.

OpenGL Object 들은 모두 상태값을 저장하는 컨테이너이다. 각각의 타입의 Object는 자신이 품고 있는 특정 상태에 의해 정의된다. 어떤 OpenGL Object는 특정 상태들의 그룹을 캡슐화하는 방식으로 사용되기도 하며 한 번의 함수 호출로 상태값들을 모두 바꾸어버리기도 한다.

Remember: 이 문서는 OpenGL이 그 스펙을 어떻게 정의하고 있는냐에 대한 것이다. 실제 드라이버에서 이런 Object가 어떻게 구현되었는가는 또 다른 문제이다. 하지만 이건 걱정할 문제는 아니고 중요한 것은 Object와 상태값이 어떻게 스펙에 정의된 대로 상호 작용하냐에 대한 것이다.

1 Object Creation and Destruction

Object를 생성하기 위해서는 정수 형태의 Object의 이름을 생성해야 한다. 이렇게 생성되는 정수값은 해당 Object에 대한 참조를 나타낸다. Object가 Context에 bind되면 처음에는 디폴트 상태값을 갖게 된다.

Object의 이름을 생성하는 함수는 glGen* 의 형태를 이룬다(*은 Object 타입의 복수형). 이런 형태의 함수들은 모두 같은 프로토타입이다. 

void glGen*(GLsizei n, GLuint *objects);

위 함수는 주어진 타입으로 n개의 Object를 생성하여 objects 파라미터에 배열로 저장한다. 이렇게 하여 한 번의 호출로 여러 개의 Object를 생성할 수 있다.

Object의 이름은 언제나 GLuint 이다. 이름은 포인터가 아니다. 이름은 레퍼런스이며 그 Object에 대한 식별자이다. 이름은 0이 아닌 32비트의 unsigned integer이다. 0인 Object는 특별한 상황에 대해 내정된 객체이다. 자세한것은 이후의 내용을 참고하자.

Legacy Note: OpenGL 3.0 이전에서는 사용자가 generation 단계를 완전히 무시하는게 가능했다. 예를 들어 사용자는 "3"이 유효한 Object 이름이라고 보고 이를 Object처럼 사용할 수 있었다. 실제 구현은 이런 식의 접근을 허용하고 "3번" Object를 처음으로 사용할 때 Object를 몰래 생성해준다. GL 3.0 에서 이러한 방식은 사용하지 않도록 권장되었다. GL 3.1 이후부터는 이런 방식이 아예 허용되지 않고있다. 아무튼 OpenGL 버전과 관계 없이 glGen* 을 통해 Object를 만드는 것이 임의로 번호를 따서 만드는 것보다 더 좋은 연습이 될 것이다.

Object를 다 사용하였다면 지워야 한다. 삭제하는 함수는 glDelete* 의 형태를 이룬다(*은 Object 타입이다). 삭제 함수들은 아래와 같은 프로토타입이다. 

void glDelete*(GLsizei n, const GLuint *objects);

삭제 함수 역시 glGen* 함수처럼 동작한다. 단, 복수의 Object를 생성하는게 아니라 지우는 것이다. 유효하지 않은 Object 값이나 0에 대해서는 무시된다.

OpenGL Object가 지워지면 이들의 이름은 더 이상 유효하지 않은 것으로 간주된다. 이렇게 삭제하고 난 다음에 바로 glGen* 을 사용하면 방금 전에 지웠던 그 이름은 다시 사용될 수도 있고 아닐 수도 있다. 그래서 이에 대해 미리 예상하고 코드를 짜서는 안된다.

1.1 Deletion unbinding

Object가 현재 Context에 bind된 상태(현재 Context 에 대해서만으로 한다)로 지워졌다면 그 Object는 그 Context에 대해 unbind될 것이다.

Note: "attachment" 가 아니라 "binding" 이다. 바인딩은 Context에 대한 것이다. attachment는 어떤 Object가 다른 것을 참조하게 되었을 때 무엇을 하느냐에 대한 것이다. attachment는 위의 상황에 대해 동일하게 동작하지 않는다.

어떤 Object들은 비정상적인 방식으로 Context와 연관되어 있다. 이런 방법이란 아래 내용을 포함한다.

  • glBindBufferRange 혹은 이와 비슷한 형태의 함수에 의해 인덱스가 지정된 Target에 bind된 Buffer Object.
  • Image 로 bind된 Texture.

위의 경우 Object가 삭제되더라도 바인딩이 해제되지 않는다. 해제가 가능한 바인딩은 해당 Object 타입의 glBind* 함수를 사용해서 바인딩이 해제되는 유형의 바인딩이어야 한다.

따라서 glBindBufferRange(GL_UNIFORM_BUFFER, …)를 사용하여 bind된 버퍼를 삭제하는 것은 마치 glBindBuffer(GL_UNIFORM_BUFFER, 0)을 호출한 것과 같은 것이다. 이렇게 하면 인덱스가 지정된 Target에서 bind를 해제하지 않는다.

1.2 Deletion orphaning

어떤 Object에 대하여 glDelete* 을 불렀다고 해서 당장 지워지는 것은 아니다. OpenGL이 애플리케이션에서 내린(issue) 커맨드를 조금 이따가 실행하기 때문에(병렬성과 퍼포먼스를 위함) Object가 실제로 지워지기 전까지는 OpenGL 서버에 남아있을 수 있다.

만일 Object가 지워진 후에도 "in use" 상태일 경우 해당 Object는 OpenGL implementation에서 alive 상태로 남게 된다.

아래와 같은 경우에는 Object가 "in use" 상태이다.

  • Object가 Context에 bind되어 있다. 이 때 Context는 꼭 현재의 것일 필요는 없다. 왜냐하면 Object를 삭제하면 삭제를 일으킨 Context에서 자동으로 unbind할 것이기 때문이다. 그래도 위의 GL 4.4 나 그 이전에 비표준적인 바인딩 포인트에 대한 경고 사항은 잘 기억해 두는게 좋다.
  • Container Object에 어태치(attach) 되어있다.

따라서 만일 Texture가 Framebuffer Object에 어태치(attach)되어 있고 Context에는 bind되어 있지 않다면 FBO(프래임버퍼 Object)는 해당 Texture를 지운 후에도 여전히 잘 동작할 것이다. FBO가 지워지거나 새로운 Texture가 어태치(attach)되어 이전 것을 대체하였을 때에만 기존 Texture가 완전히 지워질 것이다.

Object가 지워진 상태에서도 여전히 Object의 이름을 OpenGL API를 통해 알아낼 수 있다. glGetFramebufferAttachmentParameterv(GL_FRAMEBUFFER_ATTACHMENT_OBJECT_NAME) 함수는 Object의 이름을 가져온다. 그렇지만 이렇게 가져온 이름은 사용할 수 없는 것으로 간주되며 어디에도 사용하지 않는게 좋다.

Note: 어떤 환경에서는 마지막 부분을 정확하게 구현해 놓지 않았다. 따라서 진정으로 그러한 이름을 사용하지 않는게 좋다.

2 Object Usage

OpenGL의 Object는 상태값들의 모음으로 정의하고 있기 때문에 Object를 수정하려면 우선 OpenGL Context에 Object를 bind해야 한다. Object를 Context에 bind하면 그 Object 안의 상태값들을 현재 Context의 상태값이 되도록 하게된다. 이 사실이 의미하는 것은 해당 Object 내의 상태값들을 변화시키는 함수는 단순히 그 Object 안의 상태값들만을 바꾸는 것 뿐이며 이로써 다른 상태값은 보존할 수 있게된다.

새로만든 Object를 bind하면 그 Object에 대한 새로운 상태를 생성한다. 어떤 경우에는 처음으로 bind된(밑의 내용 참고) Target은 해당 Object에 대한 새로 생성된 상태의 property들에 영향을 미친다.

서로 다른 Object들은 다른 binding 함수들을 갖는다. 이런 함수들의 네이밍 방법과 공용 파라미터들은 통일되어 있다. 

void glBind*(GLenum target, GLuint object);

'*'로 표시한 부분은 Object의 타입이다. 그리고 object 는 bind되는 Object이다.

target 은 Object의 타입에 따라 또 다르다. 어떤 Object들은 Context에 대해 여러개의 Target을 가질 수 있다. 반면에 다른 Object들은 단 하나의 Target에만 bind될 수 있다. 예를 들어 하나의 Buffer Object는 Array Buffer와 Index Buffer, Pixel Buffer, Transform Buffer 혹은 또다른 Target에 bind될 수 있다.

서로다른 Target은 별개의 binding을 갖는다. 따라서 어떤 Buffer Object를 Vertex Attribute Array로 bind하고 다른 Buffer Object를 Index Buffer로 bind할 수 있다.

만일 어떤 Object가 다른 Object가 이미 bind한 Target에 bind된다면 이전에 bind된 Object는 unbind 처리될 것이다.

2.1 Object zero

OpenGL Object는 GLuint값인 0을 특별 취급한다. 그러나 어떤 Object들은 0을 또 다른 방식으로 취급한다. glGen* 함수에서 0은 리턴될 수 없다.

대부분의 Object 타입들에게 0-Object는 NULL 포인터와 가깝다. 즉, Object가 아닌 것이다. 이러한 Object 타입들에 대하여 0이 bind된다면 렌더링을 목적으로 해당 Object를 사용하는 시도는 모두 실패할 것이다.

다른 어떤 Object들에게 0-Object는 일종의 "Default Object"를 나타낸다. Texture 는 매우 복잡하지만 이러한 컨셉을 가지고 있다. 그리고 0-Texture-Object는 지울 수 없다. 이러한 Default Texture Object는 진짜 Texture Object에서 가능한 여러가지 방식으로 사용될 수 없다(예를들면 0-Texture는 FBO 에 넣을 수 없다). 따라서 0-Texture는 존재하지 않는 Texture라고 생각하는게 좋다.

Framebuffer 에서 0-Object는 Default Frambuffer 이다. Default Framebuffer는 정상적인 Framebuffer Object 와 비슷한 상태를 가지지만 아주 다른 이미지들을 갖는다1. 그리고 이러한 이미지들은 attach/detatch 될 수가 없다. 즉, 0-Framebuffer Object는 여러 FBO-specific한 인터페이스 를 지원하지 못한다. 그래도 image property query function 은 Object-0 에서도 일반적인 Framebuffer 인터페이스 함수와 같이 잘 동작한다.

Recommendation: Framebuffer Object 0의 예외적인 케이스를 포함해서 0-Object는 non-functional Object로 취급해야 한다. Object-0를 유효한 것으로 보는 Object 타입이더라도 프로그래머는 마치 그렇지 않은 것처럼 처리해야 한다. C/C++ 에서의 NULL 포인터처럼 처리하라.2

3 Multibind

몇몇 Object 타입들은 숫자가 붙여진 여러개의 특정한 Target에 bind될 수 있다. Object의 그룹을 연속된 Target에 한 번에 bind할 수 있기 때문에 유용하게 쓰인다.

중요한 사실은 이러한 multi-binding 형태는 오로지 해당 Object를 사용하는데에만 쓸 수 있고, Object를 수정하는데에는 사용할 수 없다.

multi-bind 방식의 함수와 single-bind 방식의 함수의 차이점은 다음과 같다. multi-bind 함수들은 bind시킬 Object의 배열을 받는다. 그리고 만일 이 배열의 Object들 중 하나의 Object에서 bind중 OpenGL Error 같은게 발생하여 bind할 수 없는 상황일 경우 해당 Object의 binding은 GL_INVALID_OPERATION​ 으로 실패하게 된다.

차이점은 거의 모든 다른 OpenGL 함수들은 실패하면 아무런 효과도 내지 않는데, multi-bind 함수들은 그렇지 않다는 점이다. 만일 어떤 한 Object가 무슨 이유에서건 bind할 수 없는 경우 나머지 bind가 가능한 Object들은 bind될 것이다. 오로지 그 문제있는 특정 Object의 bind만 실패하는 것이다. 그리고 Object들 중 하나가 bind할 수 없는 경우라도 에러를 생략할 것이다.

특수한 Buffer에 bind 하려다가 실패한 경우나 특정 buffer에 대해 정확하지 않은 파라미터를 넣을 경우에 대해서만 에러가 나오게 된다. 유효하지 않은 Buffer binding range 등등에 의한 에러는 Object 배열 중 어떤것도 bind시키지 않는다.

multi-bind 함수로 bind하는 Object 배열에 NULL을 넣을 수도 있다. 이 경우 해당 범위의 Target들에 대하여 0을 bind 시킨다. 이렇게 해서 배열내 모든 Object에 대한 unbind를 쉽게 할 수 있다.

multi-bind가 가능한 Object와 이들의 특별함 쓰임새는 아래와 같다. 아래 링크들은 multi-bind 함수들에 대한 자세한 정보를 담고 있다.

  • Buffer objects being bound to one of the indexed buffer binding targets.
  • Buffer objects being bound for separate attribute buffers in vertex data.
  • Textures and sampler objects being bound for use by GLSL samplers.
  • Textures being bound for use by image load/store operations.

4 Object Sharing

여러개의 OpenGL Context 를 만들 수 있다. 현재 GL Context는 스레드가 다를 경우 다른 것으로 구분된다. 보통은 각 Context는 다른 것들과 완전히 분리되어 있어 어떤 Context가 다른 Context들에 전혀 영향을 미치지 않는다.

그런데 Context를 생성하는 시점에서는 기존의 다른 Context와 Object들을 공유하는 Context를 만들 수 있다. 즉, 어떤 Context에서 생성한 Object들을 다른 Context에서 사용할 수 있다는 뜻이다.

모든 Object 타입이 Context간에 공유되는 것은 아니다. 다른 Object에 대한 참조를 가진 Object 들은 공유할 수 없다. 그 외 다른 Object 타입들은 공유할 수 있다. OpenGL Object 모델을 따르지 않는 GLSL Object 나 Sync Object 도 공유할 수 있다.

그런데 한 Context에서 Object의 상태를 수정하게 된다고 바로 다른 Context에서 그 수정 사항을 확인할 수 있는 것은 아니다. 즉, Object 상태값이 반영되는 특별한 룰 이 있다. 멀티스레드를 사용하고 있다면 어떤 Context에서 수정한 내용을 다른 Context에서 사용하기 전에 미리 동기화를 해 둘 필요가 있다.

5 Object types

Object들은 Regular Object와 Container Object의 두 카테고리로 나뉠 수 있다. 아래는 Regular Object이다.

  • Buffer Objects
  • Query Objects
  • Renderbuffer Objects
  • Sampler Objects
  • Texture Objects

Container Object는 아래와 같다.

  • Framebuffer Objects
  • Program Pipeline Objects
  • Transform Feedback Objects
  • Vertex Array Objects

6 Non-standard objects

다음은 Object 이지만 이 문서에서 서술한 OpenGL Object의 표준 동작방식3을 따르지 않는다.

  • Sync Objects
  • GLSL Shader and Program Objects
    • 여기서 Program Pipeline Object는 예외로 OpenGL Object의 표준 동작방식을 따른다.
